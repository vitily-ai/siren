# Diagnostic Address Replaces Source Coordinates on Core Diagnostics

Siren diagnostics carry a required logical `address` string (`<documentId>[:<resourceId>[:<attributeKey>]]`) instead of source coordinates on `DiagnosticBase`. `@sirenpm/core` produces diagnostics with only logical addresses and a generic `context: TContext` trivia slot; `@sirenpm/language` owns a hydrator utility that maps an address to `{ file, line?, column? }` by reading the host resource/attribute's `Origin`. `Origin` itself becomes an open discriminated base `{ kind: string; address: string }` in core, with `SyntheticOrigin` staying in core and `RangeOrigin` moving to language. We chose this over carrying file/line/column on every diagnostic (the prior shape) because coordinate plumbing was leaking source-text concerns into core's semantic surface, forcing variants like `DuplicateIdDiagnostic` to bolt on `firstLine`/`firstFile`/`secondLine`/`secondColumn` and the CLI to special-case duplicate positioning. The colon `:` is reserved in document and resource ids; violations emit a new core `E001` error at the `SirenBuilder` boundary. Parse diagnostics keep their coordinates inline via a `RangeOrigin` on the diagnostic itself, because no resource exists to hydrate from when the parser bails out. The base `DiagnosticBase<TContext>` is shared across packages, satisfying the original `diagnostic-base-trivia-slot` debt task while also decoupling core from textual positioning.
